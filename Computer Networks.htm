c)      Non Blocking I/O server program

 

Select(): To handle multiple clients using select() , it is very difficult to handle multiple file descriptor by select() because the default size to handle the file descriptor is 1024 FD . this limit the maximum FD value that can be managed by the select() function.

        Clear an fd_set

FD_ZERO(&readfds);

        Add a descriptor to an fd_set

FD_SET(master_sock, &readfds);

        Remove a descriptor from an fd_set

FD_CLR(master_sock, &readfds);

        If something happened on the master socket , then its an incoming connection FD_ISSET(master_sock, &readfds);

So Select() server program only runs for less than equal to 1024 because FD_size limit by default.

Poll(): It allows the application to pass an array of structures rather than bits. Like select poll has no such restriction.in poll, one can pass an array of poll fd structures for only FD of interest. select modifies the file descriptor set parameters. So one has to initialize the file descriptor sets for each iteration. poll has no such problem.

It cannot run in this system resource limit, because it has limited or less compatible specifications.

Poll takes large FD so it needs large storage resource, it runs for in above-core pc 4000.

 

ePoll(): It provides a functionality similar to select and poll calls. It is based on an event-driven architecture, allowing the kernel to notify the application only when there is an event on a file descriptor. This mechanism scales well even with a large number of descriptors, making it highly efficient for handling numerous connections.

This is also able to handle 4000 with less specification PC.

User Time: Higher user time could imply more intense user-level computations.

Sys Time: A high sys time might indicate that the process involves more system-level operations or kernel calls, potentially suggesting heavier I/O or system resource usage.

System Call/I/O Multiplexing & Time Metrics:

        Fork operation shows low CPU usage (% CPU) and memory consumption (%mem) compared to other operations.

        The select, poll and epoll operations show high CPU utilization with variations in memory consumption.

Real, User, System Time Metrics:

        Generally, select, poll, and epoll operations exhibit higher user and system time compared to fork and thread operations.

        The real time remains relatively consistent across different types of operations.

Conclusion:

        The variations in CPU utilization and memory consumption across different operations indicate varying resource requirements and efficiencies.

        Operations like select, poll, and epoll seem to involve more user and system time, which might be due to their nature of handling I/O multiplexing.

        Fork operation, despite having lower CPU and memory usage, could be less resource-intensive and more lightweight compared to the other operations.
